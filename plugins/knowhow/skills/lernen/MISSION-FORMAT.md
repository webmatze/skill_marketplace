# MISSION.md Format

`MISSION.md` liegt im Workspace-Root. Sie hält den _Grund_ fest, warum der User dieses Thema lernt. Jede Lehrentscheidung — was als Nächstes dran ist, welche Ressourcen aufzuschlagen sind, welche Übungen entworfen werden — soll auf dieses Dokument zurückführbar sein.

## Vorlage

```md
# Mission: {Thema}

## Warum
{1–3 Sätze. Das konkrete reale Ziel, das der User verfolgt. Was ändert sich in seinem Leben oder Job, wenn er diese Fertigkeit hat? Abstraktes wie „um X zu verstehen" vermeiden — bohre nach dem darunterliegenden Outcome.}

## Erfolg sieht so aus
- {Eine konkrete, beobachtbare Sache, die der User dann kann}
- {Noch eine konkrete Sache}
- {…}

## Rahmenbedingungen
- {Zeit, Budget, bestehende Verpflichtungen, Lernpräferenzen — alles, was den Weg begrenzt}

## Nicht im Scope
- {Angrenzende Themen, die der User aktuell explizit nicht verfolgen will — schützt die Zone der nächsten Entwicklung}
```

## Regeln

- **Eine Mission pro Workspace.** Will der User zwei unverwandte Dinge lernen, sind das zwei Workspaces.
- **Konkret vor abstrakt.** „Halbmarathon im Oktober" schlägt „fitter werden". „Eine Rust-CLI für mein Team ausliefern" schlägt „Rust lernen".
- **Gegen Vagheit Druck machen.** Wenn der User das Warum nicht artikulieren kann: erst interviewen, dann schreiben. Eine schlechte Mission ist schlimmer als keine.
- **Bei Realitätswechsel überarbeiten.** Missionen ändern sich. Wenn das Ziel des Users wandert: Datei anpassen — keine veraltete Mission künftige Sessions steuern lassen.
- **Kurz halten.** Wenn `MISSION.md` länger als eine Bildschirmseite wird, ist sie kein Kompass mehr, sondern ein Plan.
